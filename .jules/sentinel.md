## 2024-05-23 - DoS in Constant-Time Comparison
**Vulnerability:** A `constant_time_eq` implementation iterated `max(a.len(), b.len())` times, exposing a DoS vector where a large input would cause excessive CPU usage.
**Learning:** Attempts to "avoid leaking length" by checking all bytes can inadvertently introduce algorithmic complexity vulnerabilities. Standard practice is to check length first (leaking length but preventing DoS) and then compare in constant time.
**Prevention:** Prefer `subtle` crate or idiomatic constant-time comparisons that explicitly handle length checks to bound execution time.
## 2024-05-24 - Intermediate Cache Poisoning via Dynamic Origin Echoing in CORS

**Vulnerability:** The CORS middleware was dynamically reflecting the client's `Origin` header into the `Access-Control-Allow-Origin` response header without also appending `Vary: Origin`.
**Learning:** This is a classic intermediate cache poisoning vulnerability. If a caching proxy or CDN is placed in front of the application, it might cache a response allowed for one origin (e.g., `https://attacker.com`) and serve it to a subsequent request from a different origin (e.g., `https://legitimate.com`). This would bypass CORS protections because the victim's browser would receive a response with `Access-Control-Allow-Origin: https://attacker.com`. To prevent this, `Vary: Origin` must be explicitly sent whenever the origin is dynamically echoed. Also, multiple `Vary` values can be present (e.g., `Vary: Accept-Encoding`), so it's critical to use `response.headers_mut().append(header::VARY, ...)` instead of `insert()` to avoid overwriting existing headers.
**Prevention:** Whenever CORS logic involves dynamically returning the request's origin in the `Access-Control-Allow-Origin` header (as opposed to a static origin or `*`), always include a corresponding `Vary: Origin` header, and ensure `append` is used.
