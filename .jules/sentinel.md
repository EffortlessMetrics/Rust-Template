## 2024-05-23 - DoS in Constant-Time Comparison
**Vulnerability:** A `constant_time_eq` implementation iterated `max(a.len(), b.len())` times, exposing a DoS vector where a large input would cause excessive CPU usage.
**Learning:** Attempts to "avoid leaking length" by checking all bytes can inadvertently introduce algorithmic complexity vulnerabilities. Standard practice is to check length first (leaking length but preventing DoS) and then compare in constant time.
**Prevention:** Prefer `subtle` crate or idiomatic constant-time comparisons that explicitly handle length checks to bound execution time.
## 2024-05-23 - CORS Cache Poisoning via Missing Vary Header\n**Vulnerability:** When dynamically reflecting the `Origin` header in `Access-Control-Allow-Origin` responses (e.g., in axum middleware), failing to append `Vary: Origin` allows intermediate caches to incorrectly serve a cached response to a different origin, bypassing CORS.\n**Learning:** Security mechanisms that depend on request context (like CORS reflecting origins) must communicate their dependency to downstream caches via the `Vary` header to prevent cache poisoning attacks.\n**Prevention:** Always append `response.headers_mut().append(header::VARY, HeaderValue::from_static("origin"));` when dynamically generating CORS headers.
