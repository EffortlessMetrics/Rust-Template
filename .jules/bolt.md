**Learning:** Found multiple instances of synchronous file I/O (std::fs, spec loading) inside async Axum handlers. This blocks the Tokio worker thread.
**Action:** Always wrap heavy synchronous operations (like loading YAML specs) in `tokio::task::spawn_blocking` to keep the runtime responsive.

## 2025-01-20 - Optimize Security Headers Middleware
**Learning:** In Axum middleware using `axum::middleware::from_fn`, passing complex configuration structs via `AppState` often leads to per-request parsing on the hot path (e.g., repeatedly calling `HeaderValue::from_str` with raw strings inside `apply_headers`). While this isn't an algorithm change, allocating/parsing strings on every response builds up CPU and memory pressure.
**Action:** Extract expensive configuration parsing outside the middleware closure into a `Cached` representation (e.g., `CachedSecurityHeaders`) during initialization. Store `Option<HeaderValue>` instead of `String`. In the request handler, use `.clone()` on the cached `HeaderValue`s, which is practically free due to underlying `Bytes` reference counting.
