# Phase 1.3: Machine-Readable Specification & Validation

## Overview

This document describes the implementation of Phase 1.3 of the meta-contract: machine-readable specifications for xtask commands and AC reports, with automated validation to ensure implementation matches specification.

## Components Implemented

### 1. xtask Commands Specification

**File**: `/home/steven/code/Rust/Rust-Template/specs/xtask_commands.yaml`

A YAML specification defining all required xtask commands. This serves as the single source of truth for what commands must be implemented.

**Structure**:
```yaml
version: "1.0"
description: "xtask command specification for Rust Template"
commands:
  - name: <command-name>
    description: "<description>"
    required: true|false
    usage: "<usage string>"
    has_args: true|false  # optional
```

**Current Commands**:
- `check` - Format, lint, and run tests
- `bdd` - Run acceptance tests
- `ac-status` - Generate AC status report
- `bundle` - Generate LLM context bundles
- `policy-test` - Run Rego policy tests
- `quickstart` - Quick validation workflow
- `selftest` - Full CI-equivalent validation

### 2. AC Report Schema

**File**: `/home/steven/code/Rust/Rust-Template/specs/ac_report.schema.json`

A JSON Schema (draft-07) that defines the structure of Cucumber JSON reports used for AC status tracking.

**Key Features**:
- Validates the entire Cucumber JSON format
- Covers features, scenarios, tags, steps, and hooks
- Enforces required fields and valid values
- Validates step status values: `passed`, `failed`, `skipped`, `pending`, `undefined`
- Ensures AC tag format compliance

**Primary Use**: Validates that `target/ac_report.json` (generated by BDD tests) conforms to expected structure before being consumed by `ac-status` command.

### 3. Validation Module

**File**: `/home/steven/code/Rust/Rust-Template/crates/xtask/src/validation.rs`

Provides automated validation of specifications against implementation.

#### Functions

**`project_root() -> PathBuf`**
- Helper function to resolve project root from xtask crate location
- Enables tests to run from any directory

**`validate_xtask_commands_against_spec() -> Result<()>`**
- Reads `specs/xtask_commands.yaml`
- Extracts all commands marked as `required: true`
- Parses `crates/xtask/src/main.rs` to find Commands enum variants
- Validates that all required commands are implemented
- Returns error if any required commands are missing

**`normalize_command_name(name: &str) -> String`**
- Converts kebab-case to PascalCase
- Examples:
  - `"check"` → `"Check"`
  - `"ac-status"` → `"AcStatus"`
  - `"policy-test"` → `"PolicyTest"`

**`extract_commands_from_enum(content: &str) -> Result<HashSet<String>>`**
- Parses Rust source code to extract enum variant names
- Handles variants with and without fields
- Returns set of all command names found in enum

#### Test Suite

**xtask Command Validation**:
1. `test_spec_file_exists` - Ensures spec file exists
2. `test_spec_file_is_valid_yaml` - Validates YAML syntax and structure
3. `test_validate_xtask_commands` - End-to-end validation that implementation matches spec
4. `test_normalize_command_name` - Tests name conversion logic
5. `test_extract_commands_from_enum` - Tests enum parsing logic

**AC Report Schema Validation**:
1. `test_ac_report_schema_exists` - Ensures schema file exists
2. `test_ac_report_schema_is_valid` - Validates schema is valid JSON Schema
3. `test_schema_validates_sample_report` - Tests schema against minimal valid report
4. `test_ac_report_validates_against_schema` - Validates actual generated report (if exists)

## Integration Points

### 1. CI/CD Integration

The validation tests run automatically as part of `cargo test -p xtask`:

```bash
# Run validation tests
cargo test -p xtask validation

# All xtask tests (includes validation)
cargo test -p xtask
```

**Integration into CI pipeline** (recommended):
```yaml
# In .github/workflows/ci.yml or similar
- name: Validate xtask specification
  run: cargo test -p xtask validation
```

### 2. Developer Workflow

**Before adding a new xtask command**:
1. Add command definition to `specs/xtask_commands.yaml`
2. Add variant to `Commands` enum in `crates/xtask/src/main.rs`
3. Implement command handler
4. Run `cargo test -p xtask validation` to verify

**Before releasing**:
1. Run BDD tests: `cargo xtask bdd`
2. Validate report: `cargo test -p xtask test_ac_report_validates_against_schema`
3. Generate AC status: `cargo xtask ac-status`

### 3. AC Status Command Integration

The `ac-status` command (`crates/xtask/src/commands/ac_status.rs`) already uses structured JSON parsing:

```rust
// Lines 100-150: CucumberReport, CucumberFeature, CucumberElement structures
// Line 226-283: parse_cucumber_json() function reads and validates structure
```

The JSON Schema provides an independent validation layer that:
- Documents the expected format
- Can be used by external tools
- Enables schema evolution tracking
- Provides early error detection

### 4. Meta-Contract Self-Validation

This implementation creates a **self-validating system**:

```
specs/xtask_commands.yaml  ←→  Commands enum (main.rs)
        ↓                              ↓
   validation::validate_xtask_commands_against_spec()
        ↓
   Tests enforce contract
```

```
specs/ac_report.schema.json  ←→  target/ac_report.json
        ↓                              ↓
   validation::test_ac_report_validates_against_schema()
        ↓
   Schema enforces structure
```

## Usage Examples

### Validating Implementation

```bash
# Run all validation tests
cargo test -p xtask validation

# Run specific validation
cargo test -p xtask test_validate_xtask_commands

# Validate AC report schema
cargo test -p xtask test_ac_report_validates_against_schema
```

### Checking Coverage

```bash
# See what commands are defined vs implemented
cargo test -p xtask test_validate_xtask_commands -- --nocapture
```

### Schema Validation

```bash
# Validate current AC report against schema
cargo xtask bdd  # Generates target/ac_report.json
cargo test -p xtask test_ac_report_validates_against_schema
```
