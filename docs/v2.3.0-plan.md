## Epic: OTLP Tracing Implementation

**Goal:** Enable distributed tracing via OTLP (OpenTelemetry Protocol) with minimal configuration.

**Status:** Not started

### User Story

> As a service operator
> I want the service to export traces to an OTLP collector
> So that I can observe distributed request flows in production

### Acceptance Criteria

**AC-OBS-001**: OTLP export configurable via environment variables
- `OTLP_ENDPOINT` - OTLP collector endpoint (e.g., `http://localhost:4317`)
- `OTLP_SERVICE_NAME` - Service name for traces (default: `rust-iac-template`)
- `OTLP_ENABLED` - Enable/disable OTLP export (default: `false`)

**AC-OBS-002**: Tracing works with `otlp` feature flag
- Feature flag: `otlp` in `crates/telemetry/Cargo.toml`
- When disabled: Falls back to console-only tracing (current behavior)
- When enabled: Initializes OTLP exporter and layer

**AC-OBS-003**: Smoke test validates OTLP integration
- Manual test procedure documented (e.g., with local Jaeger)
- Tests that spans are exported without crashes
- Does not require full schema validation (keep it simple)

**AC-OBS-004**: Service continues to run if OTLP init fails
- Log error if OTLP endpoint unreachable or misconfigured
- Fall back to console-only tracing
- Service does not crash due to telemetry issues

### Technical Approach

**Option A: Stick with OpenTelemetry 0.31.x**
- Use latest stable version (0.31.0)
- Research API changes from 0.24/0.26
- Find working example from docs.rs or official examples

**Option B: Use OpenTelemetry 0.24/0.26**
- Use older, more stable API with known patterns
- Accept that we'll upgrade to 0.31+ in a future release
- Easier initial implementation

**Decision Point:** Research both options before committing to implementation approach.

### Deliverables

1. **Code Changes:**
   - `crates/telemetry/src/lib.rs` - OTLP initialization logic under `#[cfg(feature = "otlp")]`
   - `crates/telemetry/Cargo.toml` - OTLP dependencies with `otlp` feature flag
   - `crates/app-http/src/main.rs` - Optional: Add OTLP env var checks

2. **Documentation:**
   - `crates/telemetry/README.md` - OTLP configuration guide
   - `docs/how-to/configure-telemetry.md` - End-user guide for enabling OTLP
   - Manual smoke test procedure (e.g., with Jaeger)

3. **Tests:**
   - Smoke test demonstrating OTLP spans are exported
   - Unit tests for env var parsing (if applicable)

4. **BDD Scenario (Optional):**
   - `specs/features/telemetry.feature` - Scenario validating OTLP configuration
   - May be deferred if manual testing is sufficient

### Non-Goals

- Full schema validation of OTLP traces
- Automated integration test with OTLP collector (nice-to-have, not required)
- Metrics export via OTLP (focus on traces only)
- Custom trace attributes or resource configuration (keep it simple)

### Success Criteria

1. ✅ Service starts successfully with `otlp` feature enabled
2. ✅ Traces are exported to OTLP collector (verified manually)
3. ✅ Service falls back gracefully if OTLP fails to initialize
4. ✅ Documentation explains how to enable and configure OTLP
5. ✅ All selftest checks still pass

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| OpenTelemetry 0.31.x API is too complex | High | Fall back to 0.24/0.26, or defer to v2.4.0 |
| OTLP initialization unstable or flaky | Medium | Use feature flag, make it opt-in only |
| Smoke test too hard to document | Low | Accept simpler manual test, defer automation |
| Implementation takes >1 day | Medium | Stop and defer to v2.4.0, ship what we have |

**Guardrail:** If OTLP fights back, stop and defer. This release should be low-stress and focused.
