## Out of Scope for v2.1.0

**Deferred to v2.2.x or later:**
- New acceptance criteria or domain features
- Task Management API pilot
- Docker build automation in deploy command
- Staging/prod K8s manifest variations
- Additional how-to guides beyond adapter testing
- Performance optimization
- Additional policy areas

**Rationale:**
v2.1.0 is about **proving the infra-as-code + BDD + observability story works**. New features should wait until we have real pilot usage feedback.

## Pilot Usage (Parallel Track)

While implementing v2.1.0, run two pilots:

1. **Greenfield pilot** (internal or friendly user)
   - Start from v2.0.1 or v2.1.0-rc
   - Implement one small feature using BDD workflow
   - Document friction points

2. **Brownfield pilot** (existing project)
   - Extract `rust_iac_xtask_core` and use in real project
   - Try `xtask selftest`, `xtask bundle`, etc.
   - Document what works / what doesn't

**Friction log questions:**
- Where did the BDD flow feel heavy?
- Where did infra-as-code help vs get in the way?
- Which xtask commands are actually useful?
- What's missing from the template?

Use this feedback to drive v2.2.0 priorities, not internal speculation.

## Notes

**Why this scope?**
- v2.0.1 locked in governance + test reliability
- v2.1.0 adds the observability layer that matches the rigor of BDD + policy-as-code
- We avoid "almost done" by keeping scope tight and shippable

**Why feature-gate OTLP?**
- Not all users need OTLP (local dev, simple deployments)
- Optional feature means no breaking changes
- Users opt-in when ready

**Why /metrics matters?**
- Prometheus is industry standard
- K8s ServiceMonitor integration is expected
- Observability = BDD for runtime behavior

**Why integration tests now?**
- Adapters have been "working" but untested end-to-end
- Integration tests prove the hexagonal architecture really works
- Builds confidence for pilot usage
