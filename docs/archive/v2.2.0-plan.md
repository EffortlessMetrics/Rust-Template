# v2.2.0 Plan: Observability + Infra/BDD Depth

**Theme**: "OTLP + deeper infra-as-code/BDD polish"

**Status**: ✅ Complete (Released 2025-11-17)

---

## Guardrails

### Do NOT Touch
- Domain contracts (`business-core`, `model`)
- `rust_iac_xtask_core` public API
- High-level xtask command set

### PR Design
Each PR must:
- Keep workspace building & tested
- Include snippet of `xtask selftest` output in description
- Focus on one epic/feature
- Not introduce new domain features

---

## Epic A – OTLP Tracing (Actually Working)

**Goal**: Turn the `otlp` feature flag into a real, usable OTLP exporter.

### A1. Choose OTLP Version Strategy

**Decision Point**:
- [ ] Stick with `opentelemetry{,_sdk,-otlp} = 0.31` and implement against that API
- [ ] Downgrade to more documented version (0.24/0.26) if stability > latest

**Action**: Make explicit decision up front and stick to it for 2.2.x

**Acceptance Criteria**:
- Document chosen version in `docs/explanation/telemetry.md`
- Update `Cargo.toml` workspace dependencies if downgrading
- Research documented in decision record

### A2. Implement `init_tracing` with OTLP

**Location**: `crates/telemetry/src/lib.rs`

**Requirements**:
Under `#[cfg(feature = "otlp")]`:
- Build a `Resource` with `service.name`
- Build an exporter (tonic)
- Build a `TracerProvider` with batch processing (Tokio runtime)
- Wire a `tracing_opentelemetry` layer into subscriber

**Error Handling**:
- If anything fails (bad endpoint, network), log a warning
- Fall back to console-only tracing
- **No panics**, no propagation of OTLP errors to app code

**Acceptance Criteria**:
- `cargo build --features telemetry/otlp` succeeds
- App starts successfully with/without valid OTLP endpoint
- Graceful degradation on OTLP failures
- Unit tests for happy path and error scenarios

### A3. OTLP Integration Test / Smoke Test

**Goal**: Prove OTLP actually works end-to-end

**Setup**:
- Configure small OTLP receiver in dev (otel-collector or Jaeger demo)
- Document in `docs/how-to/test-otlp-tracing.md`

**Documentation Must Include**:
- How to run app-http with `--features telemetry/otlp`
- Required environment variables (`OTLP_ENDPOINT`, etc.)
- How to confirm spans show up in collector/Jaeger
- Screenshot or example output

**Acceptance Criteria**:
- Documented smoke test procedure
- At least one developer has successfully run full flow
- Example docker-compose or podman setup (optional)

## Epic B – Adapter Integration Tests (Proving the Hexagon)

**Goal**: At least one real integration test per adapter that's runnable (even if `#[ignore]` by default).

### B1. DB Adapter Integration Test

**Location**: `crates/adapters-db-sqlx/tests/integration.rs` (new file)

**Approach**: Choose one:
- Use `testcontainers` to spin Postgres, OR
- "Connect to `DATABASE_URL` if set; otherwise ignore the test"

**Test Flow**:
1. Migrate schema
2. `create_task` use-case (through repo)
3. `get_task`, `update_status`, `list_tasks`
4. Assert expected results

**Acceptance Criteria**:
- Test passes when `DATABASE_URL` is set (or testcontainers is running)
- Test is `#[ignore]` by default to keep CI fast
- Documented in `docs/explanation/adapters.md` ("How to run integration tests")
- At least one successful local run by developer

### B2. GRPC Adapter Smoke Test

**Location**: `crates/adapters-grpc/tests/smoke.rs` (new file)

**Minimal Test**:
1. Start GRPC server on random port
2. Use `TaskServiceImpl` with in-memory `TaskRepository`
3. Use tonic client to call `CreateTask`
4. Assert success path

**Acceptance Criteria**:
- Test is `#[ignore]` by default
- Test passes when explicitly run: `cargo test --test smoke -- --ignored`
- Documented in `docs/explanation/adapters.md`
- Demonstrates full GRPC request/response cycle

**Notes**:
- Both B1 and B2 can be ignored in normal CI to keep runs fast
- Goal is "proof that integration tests CAN be run", not "always run them"

## Epic C – LLM Ergonomics and Infra-as-Code Alignment

**Goal**: Make governance/LLM story more "useable," not just "possible."

### C1. Contextpack Metadata

**Location**: `.llm/contextpack.yaml`

**Enhancements**:
- Add optional fields:
  - `prompt` or `hint` fields for task descriptions
  - Token budget hints (even if just comments)
- Update `llm.rego` to tolerate (or validate) those fields

**Example**:
```yaml
tasks:
  implement_ac:
    description: "Context for implementing an AC"
    prompt: "Use this bundle to implement a specific acceptance criterion"
    max_bytes: 250000
    # Suggested token budget: ~60k tokens (Claude Sonnet 4.5)
```

**Acceptance Criteria**:
- Updated `.llm/contextpack.yaml` with richer metadata
- `cargo xtask bundle` still works
- Policy tests (if conftest available) validate new schema

### C2. BDD + Rego Cross-Link

**Goal**: Demonstrate policy-driven quality checks

**New BDD Scenario**:
- Location: `features/metrics.feature` (new file)
- Test `/metrics` behavior
- Steps:
  1. Start app-http
  2. Make GET request to `/health`
  3. Fetch `/metrics`
  4. Assert `http_requests_total{method="GET",path="/health"}` exists

**Optional Rego Rule** (might be 2.3.0):
- Policy to ensure new HTTP paths are covered somewhere
- This is hand-wavy, may defer to future release

**Acceptance Criteria**:
- At least one BDD scenario testing `/metrics`
- Scenario passes in `cargo xtask bdd`
- Documents metrics observability in BDD tests

## Sequencing

**Recommended order** (each is one PR):

1. **OTLP** (Epic A: A1 → A2 → A3)
   - Make decision, implement, document, smoke test

2. **DB adapter integration test** (Epic B: B1)
   - Proves persistence layer works end-to-end

3. **GRPC adapter smoke test** (Epic B: B2)
   - Proves API layer works end-to-end

4. **LLM/BDD/infra DX tweaks** (Epic C + D)
   - Polish governance story
   - Improve developer experience

**At each step**:
- Run: `cargo fmt`, `clippy`, `test`, `xtask selftest`
- Verify all 5 selftest checks pass
- Update CHANGELOG entry
- Merge to main

**After all PRs merged**:
- Final `xtask selftest`
- Update CHANGELOG with final v2.2.0 entry
- Tag `v2.2.0`
- Push tags
