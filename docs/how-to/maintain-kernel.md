# How to Maintain the Kernel

This guide is for people touching **this repo** – not forks.

The goal is simple:

> Keep the kernel healthy without guessing.

Everything here is built on three pillars:

- **spec_ledger.yaml** – canonical kernel contracts and version
- **selftest** – the only hard gate
- **docs-check** – Docs-as-Code invariants

---

## 1. Canonical sources

### Version authority

- `specs/spec_ledger.yaml -> metadata.template_version`
- `specs/spec_ledger.yaml -> metadata.last_updated`

All other version strings are *derived*:

- README, CLAUDE, ROADMAP, KERNEL_SNAPSHOT, TEMPLATE-CONTRACTS
- `specs/service_metadata.yaml`
- `specs/doc_index.yaml`
- `CHANGELOG.md`

The invariant:

```bash
cargo xtask docs-check
# must say: Version alignment... OK
```

### Governance authority

- `specs/spec_ledger.yaml` (stories, REQs, ACs, tags, must_have_ac)
- BDD features + unit tests listed under each AC
- `docs/feature_status.md` (generated by `ac-status`)

The invariants:

```bash
cargo xtask selftest
# must be green

cargo xtask ac-status
# must show all kernel ACs passing
```

> **Note:** `ac-status` will automatically run the BDD acceptance suite if the configured
> JUnit file (`target/junit/acceptance.xml`) is missing or empty, so you normally do not
> need a separate "generate reports" step to refresh `docs/feature_status.md`.

### Docs authority

- `specs/doc_index.yaml` <-> document front-matter
  - `id`, `doc_type`, `stories`, `requirements`, `acs`, `adrs`

The invariant:

```bash
cargo xtask docs-check
# doc_index <-> front-matter: OK
# feature_status header: OK
```

## 2. Daily maintenance commands

From the repo root, inside `nix develop`:

```bash
# Quick "am I safe to touch this?" check
cargo xtask kernel-status

# Before pushing a kernel change
cargo xtask docs-check
cargo xtask selftest

# When changing specs/tests/docs
cargo xtask ac-status
```

Interpretation:

- **`kernel-status`**
  - You should see all subsystems green.
  - If something is red, the output tells you exactly which command to run next.

- **`docs-check`**
  - Fails on:
    - version drift across docs,
    - doc_index <-> front-matter mismatch,
    - feature_status header not matching spec_ledger.

- **`selftest`**
  - Fails on:
    - kernel AC test failures,
    - governance graph invariants,
    - policy tests,
    - devex contract.

## 3. Safe version bump workflow

When you want to bump the kernel from `3.4.0` -> `3.4.1`:

1. **Update spec_ledger metadata**

   ```yaml
   # specs/spec_ledger.yaml
   metadata:
     schema_version: "1.0"
     template_version: "3.4.1"
     last_updated: "2025-12-01"
   ```

2. **Let automation update consumers**

   ```bash
   nix develop
   cargo xtask release-prepare 3.4.1
   ```

3. **Validate Docs-as-Code**

   ```bash
   cargo xtask docs-check
   # Version alignment... OK
   ```

4. **Validate governance**

   ```bash
   cargo xtask selftest
   ```

5. **Tag the kernel**

   ```bash
   git tag "$(cargo xtask version --json | jq -r .kernel_tag)" -m "Kernel closure"
   ```

6. **Push**

   ```bash
   git push origin main --tags
   ```

## 4. Adding / changing kernel ACs

### 4.1 Add a new kernel AC

1. **Add to `spec_ledger.yaml`** under the right REQ
2. **Add tests**:
   - BDD: `specs/features/*.feature` with `@AC-XXX`
   - Unit: in the appropriate crate + mapped in `tests:` list
3. **Run the ladder**:

   ```bash
   cargo xtask ac-status
   cargo xtask docs-check
   cargo xtask selftest
   ```

If `ac-status` shows the new AC as `[UNKNOWN]`, it means your tests aren't wired properly.

### 4.2 Demoting / changing `must_have_ac`

If you want to change `must_have_ac` on an AC:

- Treat it as a **kernel contract change**.
- Follow the existing guide:
  - `docs/how-to/change-acceptance-criterion.md`
  - ensure `AC-PLT-AC-DEMOTION-GOVERNED` is satisfied.

## 5. When something is red

If **kernel-status** or **selftest** fails, triage in this order:

1. **Specs**
   - Broken YAML? Wrong IDs? Run:

     ```bash
     cargo xtask ac-status
     ```

2. **Docs-as-Code**
   - Version mismatch? Run:

     ```bash
     cargo xtask docs-check
     ```

3. **Tests**
   - BDD or unit failures? Use:

     ```bash
     cargo xtask test-changed
     cargo xtask test-ac AC-XXX
     ```

4. **CI-only gates**
   - Example fork, policy tests, etc. may show up as CI failures but not block local selftest.
   - Check `.github/workflows/` and the relevant `AC-*` in `spec_ledger.yaml`.

## 6. Summary

For kernel maintainers:

- **Single health command:** `cargo xtask kernel-status`
- **Canonical truth:** `specs/spec_ledger.yaml`
- **Gates that matter:** `docs-check`, `selftest`, `ac-status`

If all three are green, the template is healthy. Everything else (forks, pilots, Knowledge Hub) happens in other repos.

---

## 7. Kernel mental model

For the high-level system architecture, see `docs/explanation/architecture.md` and `docs/explanation/TEMPLATE-CONTRACTS.md`.

In brief:

- **Specs layer**: `spec_ledger.yaml`, `tasks.yaml`, `devex_flows.yaml` define contracts.
- **Runtime layer**: `spec-runtime` crate loads and validates specs.
- **Surfaces**: `xtask` (CLI), `app-http` (HTTP + `/platform/*` APIs), `/ui` (web dashboard).

Key contracts are enforced by `selftest` and documented in `TEMPLATE-CONTRACTS.md`.

## 8. Handover checklist (for the next maintainer)

You're in good shape if you can answer "yes" to:

- [ ] I can run `nix develop` + `cargo xtask kernel-status` and interpret the output.
- [ ] I know that `specs/spec_ledger.yaml` is the **canonical version** and governance source.
- [ ] I know `docs-check` enforces:
  - version alignment,
  - doc_index ↔ frontmatter sync,
  - feature_status header invariants.
- [ ] I know `selftest` enforces:
  - kernel AC health,
  - devex & graph invariants,
  - policy tests,
  - BDD harness semantics (via `bdd::is_bdd_success`).
- [ ] I know how to add/change ACs in `spec_ledger.yaml` and wire appropriate tests.
- [ ] I know how to bump the template_version safely and tag a new `vX.Y.Z-kernel`.
- [ ] I know where this maintainer runbook is: `docs/how-to/maintain-kernel.md`.
- [ ] I know how to keep VS Code / rust-analyzer in sync with Nix (`nix develop && code .`).

If all of that is true, you can take this over without tripping invisible wires.
